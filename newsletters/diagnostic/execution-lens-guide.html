Execution Lens Framework

This assessment evaluates how your operations actually perform across seven core execution lenses.

Each lens reflects how work really gets done across bidding, projects, field execution, and closeout.

How to Read Your Score

Each section is scored out of 10 points.

9–10 → Installed System (Controlled Execution)
You have defined rules. Work is processed consistently. Execution is predictable.
5–8 → Partial System (Inconsistent Execution)
Some structure exists, but execution varies. People fill the gaps.
0–4 → No System (Reactive Execution) ⚠️
No defined rules. Work relies on reaction, follow-up, and individual effort.
01

Process

Score: __ / 10
What This Measures

Whether your operation has clearly defined processing rules for how work moves from:

Input → Action → Outcome

Or if work is handled differently each time.

What This Looks Like in Practice

Strong Process:

  • Every RFI follows the same path
  • Every project kickoff follows the same structure
  • Every bid moves through defined stages

Weak Process:

  • "It depends on the PM"
  • Steps are skipped or reordered
  • Workflows live in people's heads
Current State
Typical Reality

Methodology: Work flows are often "person-to-person" or managed through decentralized folder structures like Dropbox.

Consistency: Each Project Manager (PM) utilizes a unique, non-standardized method for managing their specific jobs.

Key Breakdowns
  • Handoffs: Critical project data is frequently lost during the transition from sales/pre-construction to operations
  • Duplication: Staff perform "double entry" by manually moving data from emails into tracking spreadsheets
  • Inconsistency: No central process to capture information consistently
Desired State
What to Build

Templatization: Automatic creation of standardized project structures based on project type (e.g., "Run" vs. "Strategic").

Connectivity: A "single source of truth" where every project phase, from design to shipping, is linked in a continuous flow.

Next Steps (Based on Your Score)
High Priority Actions
  • Map your top 5 workflows (bid → kickoff → execution → closeout) — document what actually happens, not what should happen
  • Create templates for each project type so new projects auto-populate with required phases and documents
  • Define handoff rules — specify exactly what data transfers between phases and how
  • Eliminate manual data entry — integrate systems so information moves automatically between phases
Resource / Exercise

Write down:

  • The last 3 projects you ran
  • Step-by-step what actually happened

That—not what you think—is your current process.

02

Accountability

Score: __ / 10
What This Measures

Whether ownership is built into your system—or depends on follow-up.

What This Looks Like in Practice

Strong Accountability:

  • Every task has a clearly assigned owner
  • Ownership is visible and tracked
  • Work doesn't move without responsibility

Weak Accountability:

  • "The team is handling it"
  • Tasks sit until someone follows up
  • Ownership changes depending on the situation
Current State
Typical Reality

Ownership: Tasks often sit in "someone's court" (e.g., "Sarah's court" for budgeting) without visibility for the rest of the team.

Enforcement: Follow-ups on outstanding RFIs or submittals are handled manually through repetitive emails or phone calls.

Key Breakdowns
  • Stagnation: RFIs stall for weeks because there is no clear owner or automated reminder
  • Ambiguity: PMs may oversee construction but have no accountability for project budgets
  • Manual Follow-up: Leadership spends time chasing instead of managing
Desired State
What to Build

Automated Assignment: Logic that automatically assigns an architect, engineer, or PM to a task the moment it is created.

Gatekeeping: Systems that prevent a project from moving to the next phase until a specific owner signs off on the previous gate.

Next Steps (Based on Your Score)
High Priority Actions
  • Define ownership rules — for each task type, specify who is responsible by default
  • Create automated assignments — when an RFI is created, automatically assign to the appropriate discipline
  • Set escalation thresholds — "If RFI > 5 days, escalate to PM" or "If budget variance > 10%, escalate to controller"
  • Build visibility into ownership — make it obvious who owns what and what is overdue
Resource / Exercise

Take one workflow (RFIs, submittals, etc.):

For each step, answer:

"Who is responsible for this being completed?"

If you hesitate—you found a gap.

03

Failure Mode

Score: __ / 10
What This Measures

Whether your system detects problems early—or relies on people to notice them.

What This Looks Like in Practice

Strong Failure Detection:

  • Aging RFIs are flagged automatically
  • Schedule delays are visible early
  • Issues are surfaced before impact

Weak Failure Detection:

  • Problems discovered during meetings
  • Issues identified after delays occur
  • Reliance on escalation
Current State
Typical Reality

Detection: Issues are often discovered late (e.g., "in the fourth quarter") because leadership lacks a real-time snapshot of what is off-track.

Mitigation: When a delay occurs (e.g., equipment lead times of 52 weeks), there is no automated "trickle-down" to update downstream fabrication or shipping schedules.

Key Breakdowns
  • Information Silos: External emails from suppliers about ship-date changes are not integrated into the schedule
  • Compounding Delays: One missed task stalls an entire project because dependency logic is only in a manager's head
  • Crew Impact: Crews arrive at sites without materials due to lack of integrated tracking
Desired State
What to Build

Proactive Alerts: Automated notifications sent to PMs when a task is marked "Stuck" or a due date passes without completion.

Dynamic Scheduling: Gantt charts with dependencies that automatically push all following tasks when an early-stage item is delayed.

Next Steps (Based on Your Score)
High Priority Actions
  • Define status thresholds — set specific days (e.g., "5 days without update = stuck") that trigger alerts
  • Build dependency logic — map which tasks block which phases so delays cascade automatically
  • Create "Stuck" status — flag items that are delayed or at risk, forcing resolution before the project moves forward
  • Automate notifications — when a deadline passes, notify the owner and escalate to their manager
Resource / Exercise

Ask:

"How would I know something is off—without asking anyone?"

If the answer is unclear → no detection system exists.

04

People Dependency

Score: __ / 10
What This Measures

How much your operation depends on specific individuals vs system-driven execution.

What This Looks Like in Practice

Low Dependency:

  • Work continues regardless of who is present
  • Knowledge is captured in systems
  • Processes are transferable

High Dependency:

  • "Only X knows how to do this"
  • Work stalls when people are unavailable
  • Knowledge lives in experience, not systems
Current State
Typical Reality

Knowledge Base: Information is frequently stored in "people's heads" or individual desktop folder systems.

Bottlenecks: Specific individuals become operational constraints because they are the only ones with knowledge or access.

Key Breakdowns
  • Siloed Data: Global accounts managed with different naming conventions across regions
  • Retirement Risk: Critical knowledge held by "seasoned people" who are retiring
  • Onboarding Drag: New staff take months to become productive because knowledge isn't documented
Desired State
What to Build

Role-Based Access: Standardized views where any team member can step into a project and understand its status by looking at the "metadata".

Centralized Communication: All correspondence and files tied to a specific item so project history is not lost if a PM leaves.

Next Steps (Based on Your Score)
High Priority Actions
  • Document workflows — capture decision rules and process logic so others can execute them
  • Standardize naming — ensure global accounts, projects, and items use consistent naming across all regions
  • Centralize communication — move all project correspondence into the system, not individual emails
  • Create role-based views — ensure anyone stepping into a role can see exactly what they need to know
Resource / Exercise

Ask:

"If this person left tomorrow, what breaks?"

That's your dependency risk. List it. Systematize it.

05

Visibility

Score: __ / 10
What This Measures

Whether you can see what's happening in real time—or rely on updates and communication.

What This Looks Like in Practice

High Visibility:

  • Project status visible instantly
  • Tasks, budgets, and timelines accessible
  • No need to ask for updates

Low Visibility:

  • Status requires meetings or calls
  • Information is scattered
  • Teams operate with partial information
Current State
Typical Reality

Reporting: Management relies on monthly manual roll-ups that are "strenuous to keep up to date" and often "clunky".

Granularity: Current tools may offer a high-level view but fail to provide the ability to "drill down" into the specific task causing a delay.

Key Breakdowns
  • Blind Spots: Leadership lacks visibility into "who has capacity" to take on new work
  • Lag Time: Decisions are based on "reports" rather than "reality"
  • Drill-Down Inability: Can't easily identify which specific task is causing a delay
Desired State
What to Build

Live Dashboards: Real-time visualizations of pipeline forecasting, project spend, and "items ready for production".

Filtered Views: Custom interfaces for different users (e.g., "friendly" shop-floor views versus detailed PM "metadata" tables).

Next Steps (Based on Your Score)
High Priority Actions
  • Build a portfolio dashboard — 30,000-foot view showing all active projects, status, and pipeline value
  • Create drill-down capability — click a project to see tasks; click a task to see details and owner
  • Add capacity views — show which teams/individuals have capacity and who is overallocated
  • Eliminate manual reporting — all metrics should pull directly from your system, not spreadsheets
Resource / Exercise

Ask:

"Can I see the status of every active project in under 30 seconds?"

If not → visibility gap. That's your priority.

06

Cost / Cash

Score: __ / 10
What This Measures

How operational inefficiencies translate into financial impact.

What This Looks Like in Practice

Controlled Cost:

  • Delays and inefficiencies are tracked
  • Financial impact is visible
  • Decisions are tied to cost

Uncontrolled Cost:

  • Time loss is untracked
  • Delays go unnoticed financially
  • Margins erode without visibility
Current State
Typical Reality

Accounting: Financial data lives in disparate systems (NetSuite, Workday, QuickBooks) and must be manually reconciled with project spreadsheets.

Tracking: Costs are decoupled from progress; budgets are "kept over in Sarah's court" where PMs cannot see them.

Key Breakdowns
  • Margin Leakage: Quotes honored a year later without price escalation because scope/date not tracked
  • Documentation Lag: Invoicing delayed by weeks because closeout documentation must be compiled manually
  • Hidden Time Loss: Time spent on rework, delays, and coordination is not costed or tracked
Desired State
What to Build

Job Costing Integration: Automated feeds from accounting software that update "Actual vs. Estimated" costs on project boards.

Real-time Time Tracking: Mobile apps where field crews log hours against specific CSI codes, feeding directly into project budget roll-ups.

Next Steps (Based on Your Score)
High Priority Actions
  • Integrate accounting systems — connect job costing to project tracking so "Actual vs. Estimated" is always visible
  • Enable mobile time tracking — field crews log hours against project tasks, feeding directly into budgets
  • Link billing to execution — automate the trigger for invoicing when tasks are marked "complete" and signed off
  • Measure cost of delays — calculate financial impact of schedule slips and inefficiencies
Resource / Exercise

Ask:

"Where are we losing time—but not measuring it?"

That's where margin is leaking. Quantify it.

07

Scalability

Score: __ / 10
What This Measures

Whether your operation can handle increased volume without breaking.

What This Looks Like in Practice

Scalable:

  • Systems handle increased workload
  • Workflows remain consistent
  • Teams don't rely on hero effort

Not Scalable:

  • More work = more chaos
  • Teams become overloaded
  • Quality drops
Current State
Typical Reality

Volume Limits: Managers are limited by the number of accounts they can manually track (e.g., 30 accounts is a "mature" limit for a salesperson without a CRM).

Friction: Increasing project volume leads to "monster boards" with too many columns that "just don't work inherently" under load.

Key Breakdowns
  • Manual Bottlenecks: Every new project requires hours of manual setup in Excel or MS Project
  • System Fragmentation: Growth leads to more "siloed" modules that don't talk to each other
  • Quality Degradation: Under volume, processes break and teams resort to ad-hoc solutions
Desired State
What to Build

System Leverage: Automations that handle handoffs between departments automatically, allowing teams to "move a lot faster" with less manual work.

Templatized Growth: The ability to "copy-paste" complex project structures (like substations) to handle hundreds of similar jobs simultaneously.

Next Steps (Based on Your Score)
High Priority Actions
  • Build project templates — create reusable structures for each project type so new projects auto-populate
  • Automate handoffs — eliminate manual work between departments through system integrations
  • Reduce manual setup — one click should create a complete project structure, not hours of work
  • Test under load — deliberately increase volume to identify what breaks and fix it proactively
Resource / Exercise

Ask:

"If we doubled our workload tomorrow, what breaks first?"

Fix that first. That's your constraint.

🔥 Final Note (To Tie It All Together)

This isn't about fixing isolated issues.

It's about installing a system where:

Inputs are processed consistently, without relying on memory, follow-up, or individual effort

Right now, most operations run on:

Input → Reaction → Outcome

The goal is:

Input → Rule → Action → Controlled Outcome

Your Next Move:

Find your lowest-scoring lens. That's your constraint. Start there. Fix that lens, and you create capacity to improve the others. The lenses work together—strengthen one, and the others become easier to fix.
